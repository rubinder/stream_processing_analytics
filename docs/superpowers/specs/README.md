# Design specs

One Markdown file per design decision. Filename: `YYYY-MM-DD-<topic>-design.md`.

Each spec covers locked decisions, architecture, schemas, runtime policies (late data, error handling), testing strategy, and out-of-scope items. Read the spec to understand **what** the system does and **why**; read the source to understand the current **how**.

If a fundamental design choice changes (e.g., switching from broadcast state to external lookups), update the spec or write a follow-up dated spec — don't bury the change in code.
